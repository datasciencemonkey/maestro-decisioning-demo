This is workflow durability, not conversation checkpointing. The distinction is fatal to Paths 2 and 3.

Path 2 — pydantic-deepagents: Why It Doesn't Fit

Three Fatal Problems

1. Not durable execution. Checkpoints are conversation snapshots stored on local disk (.pydantic-deep/sessions/{id}/). No ACID guarantees — if the process dies mid-checkpoint, state corrupts. No scheduled wake-up mechanism exists. You would still need DBOS for Durable Execution.

2. Breaks the governance story. State lives on local filesystem, outside Unity Catalog. The "all durable agent state lives inside Databricks" thesis — the structural moat of the demo — collapses.

3. No MLflow integration. Uses Logfire for tracing. Would require ripping out the verified MLflow setup. v0.3.1 with known security vulnerabilities including command injection and URI exposure.

The fundamental mismatch:
pydantic-deepagents solves: "How do I manage a long-running AI conversation with checkpoints?"
Maestro needs: "How do I persist a business decision durably, wake up 11 hours later, and act on it — all inside Databricks?"
These are fundamentally different problem domains.

Recommendation

PATH 1 — Stay the Course

Pydantic AI + Skills + DBOS on Lakebase + MLflow Tracing

The requirement is workflow durability, not conversation checkpointing. The demo's core thesis is the governance plane. Infrastructure is verified. Simplicity wins for a 15-minute demo.

Five Reasons (CEO lens)

  1. Workflow durability ≠ conversation checkpointing. DBOS provides ACID state + durable sleep with automatic wake-up. pydantic-deepagents checkpoints conversations — different abstraction.
  2. The demo's core thesis is the governance plane. "All durable agent state lives inside Databricks." State on local files undermines this entirely.
  3. Infrastructure is verified. Switching means redoing weeks of validation work against v0.3.x packages.
  4. Simplicity wins. Agent Reasoning uses a single supervisor. It needs 9 tools organized as skills, durable state, a beautiful trace. Path 1 delivers all with minimal parts.
  5. Extensibility built-in. Skills are in Phase A from day one, packaging domain best practices. Sub-agents and context compression layer on for Phase B/C without replacing the foundation.

Nice-to-Have: Long-Running Agents (Phase B/C — Additive)

  • pydantic-ai-summarization for context compression — layer on top, don't replace DBOS
  • Custom conversation checkpointing to Lakebase — store history in a table, not files
  • DBOS step functions for multi-step workflows with per-step durability
Final word: Features without architectural alignment introduce unnecessary complexity. The question is not "what does it offer?" but "does it solve our specific problem?" Our problem: durable business state inside the Databricks governance plane. Path 1 solves it. Paths 2 and 3 do not.

Recommendation: implement Path 1, build Agent Reasoning, and prioritize simplicity.

Pydantic AI Native Multi-Agent (Built-in)

Pydantic AI natively supports two multi-agent patterns without any additional packages:

Agent Delegation

One agent calls another inside a tool. Supervisor/worker pattern with shared deps and cost aggregation.

@supervisor.tool
async def analyze_campaigns(
  ctx: RunContext[CDPDeps],
  customer_id: str
) -> CampaignAnalysis:
  r = await campaign_agent.run(
    f"Analyze {customer_id}",
    deps=ctx.deps,
    usage=ctx.usage,  # aggregate cost
  )
  return r.output

Programmatic Hand-off

Application code decides which agent runs next. Useful for multi-step workflows with human-in-the-loop.

# Application code orchestrates
result1 = await reasoning_agent.run(
  "Analyze cart abandonment",
  usage=usage,
)
# Hand off to activation agent
result2 = await activation_agent.run(
  f"Execute: {result1.output}",
  usage=usage,
)

Limitations

Native approach is sufficient for Agent Reasoning but has gaps for complex orchestration:

  • No async background tasks
  • No dynamic agent creation
  • No built-in task lifecycle
  • No progressive disclosure
  • Deps must be same type or subset

pydantic-ai-skills: Domain Expansion Without Code Changes

Skills solve the extensibility problem: how do you go from 9 tools to 50+ without rewriting the agent? Skills are modular domain packages with instructions, resources, and executable scripts.

Why Skills Matter for Maestro

DimensionWithout SkillsWith Skills (Phase A)
Add new domainWrite @agent.tool, update system prompt, redeployDrop a folder in skills/, agent auto-discovers
Token overheadAll 9 tool schemas always loadedProgressive disclosure: load details only when needed
DocumentationDocstrings + external docsSKILL.md is both agent instruction and developer doc
SharingCopy Python functionsGit-based registries, composable across agents
Scale to 50 domainsUnwieldy system promptAgent sees skill names; loads details on demand

How it works: 4 tools are added to the agent — list_skills, load_skill, read_skill_resource, run_skill_script. Skills live as directories with a SKILL.md and optional scripts/resources. The agent discovers available skills via metadata in the system prompt, then loads full instructions only when a task requires that domain.
