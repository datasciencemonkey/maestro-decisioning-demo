Codebase Explainer — Agent Loop

Maestro CDP — Codebase Explainer

How the agent loop works, step by step. A guided walkthrough of the reasoning agent, copywriter agent, and DBOS workflow that orchestrates them.

Audience: You're new to the codebase and want to understand how the agent loop works. This document walks you through the actual code — no pseudocode, real imports, real function signatures, real prompts.

1. The Big Picture

When a customer abandons their cart, a single DBOS workflow fires. Inside that workflow, two Pydantic AI agents do the thinking — one decides what to do, the other writes the email. Here's what happens:

1
CartAbandonedEvent fires (from the frontend or event stream)
2
Agent 1 (Reasoning) wakes up, reads 8 tools about customer, cart, campaigns, feasibility, timing
3
Agent 1 produces a DecisionArtifact — what to do, why, and when
4
DBOS persists the decision to Lakebase, then sleeps (simulating the optimal send window)
5
DBOS wakes up, re-evaluates context (check for changes), then hands off to Agent 2
6
Agent 2 (Copywriter) receives the decision artifact and writes the email
7
DBOS simulates sending the email and updates journey status to completed
Why two agents? Agent 1 is a reasoner — it considers constraints, makes trade-offs, decides next steps. Agent 2 is a writer — it takes that decision and makes it human and personal. Separating concerns keeps both agents focused and testable.

2. How Agent 1 (Reasoning) Is Built

Agent 1 lives in src/maestro/agent.py. It's a single Pydantic AI agent that reasons across 9 domains (customer profile, cart, campaigns, feasibility, frequency caps, support tickets, propensity scores, send timing, and journey persistence). Let's walk through how it's constructed.

2a. The MaestroDeps Dataclass — Dependency Injection

Every tool in the agent receives a RunContext that includes dependencies. Here's what gets injected:

@dataclass
class MaestroDeps:
    """Runtime dependencies injected into every tool via RunContext."""

    customer_id: str
    event: CartAbandonedEvent
    db_url: str | None = None
The customer_id and event are passed when the agent is created. Every @agent.tool decorated function receives a ctx: RunContext[MaestroDeps] parameter that lets it access these dependencies without global state.

2b. The System Prompt — What Agent 1 Is Told

The system prompt is the agent's constitution. It's long (100+ lines), but here's the key sections:

SYSTEM_PROMPT = """\
You are the agentic CDP disposition engine for Fluttershy, a print-on-demand
photo product brand. A cart_abandoned event has fired.

YOUR GOAL: Decide how to engage this customer for cart recovery, balancing:
  • Their context (profile, cart, support history)
  • Operational constraints (production calendar, frequency caps, consent)
  • Likelihood of conversion (propensity score)
  • Optimal timing (send-time model + quiet-hours policy)

REASONING APPROACH — this is critical:
  1. Start by understanding WHO the customer is (profile) and WHAT they left
     behind (cart). These inform every subsequent decision.
  2. Check operational constraints — can we even fulfill this order? Are there
     campaign conflicts? Would another send breach frequency caps?
  3. Assess likelihood and timing — how likely is recovery? When should we act?
  4. YOU DECIDE the order of tool calls based on what you learn. When a tool's
     output makes a subsequent tool unnecessary, SKIP it and explain why.
  5. When two signals contradict (e.g., customer is in an active campaign AND
     frequency cap would breach), reconcile them explicitly — state the
     conflict, your resolution, and why.

DECISION OUTPUT:
When you have enough information, produce a DecisionArtifact with:
  • verdict: your overall recommendation (e.g., "re-prioritize", "suppress",
    "proceed", "defer")
  • decisions: specific actions — each with type, target/value, and reason:
    - suppress_from: remove customer from a lower-priority campaign
    - prioritize_in: elevate a campaign for this customer
    - tone: communication style (warm/personal, professional, empathetic)
    - send_time: when to send, with timezone
    - channel: communication channel (email, sms, push)
  • contributing_signals: each signal that influenced your decision, with weight
  • rationale: one paragraph, human-legible, explaining your reasoning chain

...
CDP DOMAIN KNOWLEDGE:
  Campaign Priority: triggered > active/scheduled > paused > dormant
  Frequency Cap Resolution: suppress lower-priority campaign, never the triggered one
  Quiet Hours: 9 PM – 7 AM in customer timezone; shift to next 8 AM
  Tone: pet owner → warm/personal with pet name; repeat buyer → familiar; new → welcoming
"""
Notice the prompt emphasizes reasoning, not scripting. It tells Claude to skip tools when unnecessary ("when a tool's output makes a subsequent tool unnecessary, SKIP it"), to reconcile contradictions, and to explain trade-offs. This is the opposite of a rigid function-calling pipeline.

2c. Agent Construction

The agent is created with Pydantic AI's builder pattern:

def create_agent(model) -> Agent[MaestroDeps, DecisionArtifact]:
    """Create the Maestro CDP agent with all tools registered."""

    maestro_agent = Agent(
        model,
        output_type=DecisionArtifact,
        deps_type=MaestroDeps,
        instructions=SYSTEM_PROMPT,
        name="maestro-cdp",
        retries=1,
        capabilities=[SkillsCapability(directories=["./app_skills"])],
    )

    # ... tools registered below ...

    return maestro_agent
Key parameters:
  • output_type=DecisionArtifact — Claude must output structured JSON that matches this Pydantic model. If the LLM returns something that doesn't parse, Pydantic AI retries.
  • deps_type=MaestroDeps — every tool receives ctx: RunContext[MaestroDeps] to access customer ID and event.
  • instructions=SYSTEM_PROMPT — the reasoning constitution above.
  • retries=1 — if Claude's first output doesn't match the schema, try once more.

2d. Tool Registration — The @agent.tool Decorator

Tools are async functions decorated with @maestro_agent.tool. Here are two examples:

    @maestro_agent.tool
    async def read_profile(
        ctx: RunContext[MaestroDeps], customer_id: str
    ) -> Profile:
        """Read the customer profile — demographics, pet info, timezone,
        channel preferences, consent status, and loyalty tier."""
        if customer_id in PROFILES:
            return PROFILES[customer_id]
        raise ValueError(f"Unknown customer: {customer_id}")

    @maestro_agent.tool
    async def check_frequency_cap(
        ctx: RunContext[MaestroDeps],
        customer_id: str,
        channel: str,
        window_days: int,
    ) -> CapStatus:
        """Check if sending on this channel would breach the frequency cap.
        Returns current count, queued count, cap limit, and breach status."""
        key = (customer_id, channel)
        if key in CAP_STATUSES:
            return CAP_STATUSES[key]
        return CapStatus(
            customer_id=customer_id, channel=channel,
            cap=3, current=0, queued=0, status="ok", window_days=window_days,
        )
Every tool is:
  • Async — Pydantic AI runs them concurrently when possible.
  • Documented — the docstring tells Claude what the tool does. Claude reads these docs before deciding which tool to call.
  • Typed — input and return types are explicit. Pydantic AI validates inputs before calling the tool.
  • Injectable — the ctx: RunContext[MaestroDeps] parameter provides access to customer ID, event, and database URL without global state.

There are 8 more tools in the full agent: read_cart, check_production_feasibility, list_active_campaigns, read_support_tickets, score_propensity, optimal_send_time, and persist_journey_state. All follow the same pattern.

2e. The Runner — run_maestro()

The runner is the entry point. It's called from the API or workflow:

async def run_maestro(
    event: CartAbandonedEvent,
    model,
    db_url: str | None = None,
) -> DecisionArtifact:
    """Run the Maestro agent on a cart_abandoned event.

    Returns a structured DecisionArtifact with verdict, decisions,
    contributing signals, and rationale.
    """
    agent = create_agent(model)
    deps = MaestroDeps(
        customer_id=event.customer_id,
        event=event,
        db_url=db_url,
    )

    prompt = (
        f"Cart abandoned event for customer {event.customer_id}. "
        f"Cart ID: {event.cart_id}. "
        f"Items: {[item.model_dump() for item in event.items]}. "
        f"Total: ${event.cart_total:.2f}. "
        f"Abandoned at: {event.abandoned_at.isoformat()}. "
        f"Evaluate this customer for cart recovery disposition."
    )

    result = await agent.run(prompt, deps=deps)
    return result.output
The runner:
  • Creates the agent (which registers all tools).
  • Builds the MaestroDeps with customer ID and event data.
  • Constructs a short prompt that references the event and asks the agent to decide.
  • Calls await agent.run(prompt, deps=deps) — this is where Claude reasons.
  • Returns the parsed DecisionArtifact.
What happens inside agent.run()? Claude reads the system prompt, considers the tools, decides which ones to call, calls them (asynchronously), sees the results, and repeats until it has enough information. Then it outputs a JSON structure that Pydantic AI parses into a DecisionArtifact object.

3. How Agent 2 (Copywriter) Is Built

Agent 2 lives in src/maestro/email_agent.py. It's simpler than Agent 1 — no tools, no branching logic. It receives context (customer profile, cart, tone decision) and writes a short, branded email. It's a pure text generation task.

3a. The System Prompt

COPYWRITER_PROMPT = """\
You are a DTC email copywriter for Fluttershy, a premium pet photo products brand.

Write a warm, personal cart recovery email. Rules:
  - Subject line: under 50 characters, mention the pet by name if available
  - Body: under 100 words, conversational, mention the specific product left behind
  - Tone: {tone} — match the customer's relationship with the brand
  - Use the pet's name naturally (not forced). If no pet, focus on the product.
  - Include one clear CTA. Never use urgency tactics, guilt, or countdown timers.
  - The hero image URL is: {hero_image_url}
  - CTA URL is: https://fluttershy.com/cart/resume?cid={customer_id}

Write the body_html as a simple HTML fragment (no full document, no  tags).
Use 
 for greeting, 
 for body, styled simply. The email template wrapper
handles the outer chrome.
"""
This prompt is prescriptive — it gives hard rules (50 char subject, 100 word body, specific URLs) and tone constraints. Unlike Agent 1's reasoning prompt, this one says "do this, not that."

3b. CopywriterDeps — What Context Agent 2 Receives

@dataclass
class CopywriterDeps:
    customer_id: str
    profile: Profile
    cart: Cart
    tone: str
    hero_image_url: str
Agent 2 doesn't need to fetch anything. It receives everything it needs as dependencies:
  • profile — customer name, pet name, loyalty tier (for tone calibration).
  • cart — the abandoned items (to mention specifically in the email).
  • tone — extracted from Agent 1's decision (e.g., "warm + personal").
  • hero_image_url — static (e.g., "/whiskers.jpg"), but could be dynamic per customer.

3c. Agent Construction

def create_copywriter_agent(model) -> Agent[CopywriterDeps, EmailContent]:
    """Create the copywriter agent."""
    return Agent(
        model,
        output_type=EmailContent,
        deps_type=CopywriterDeps,
        instructions=COPYWRITER_PROMPT,
        name="fluttershy-copywriter",
        retries=1,
    )
Same pattern as Agent 1: structured output (EmailContent), dependency injection, system prompt, retries. But no tools — Claude just reads the context and writes.

3d. Building the Prompt and Running Agent 2

async def compose_email(
    artifact_json: str,
    model,
) -> EmailContent:
    """Compose a personalized email from the decision artifact context.

    Uses synthetic data to look up profile/cart, then runs the copywriter agent.
    """
    from maestro.synthetic import CARTS, PROFILES

    artifact = json.loads(artifact_json)
    customer_id = artifact.get("customer_id", "cust_88241")
    profile = PROFILES.get(customer_id)
    cart = CARTS.get(customer_id)

    if not profile or not cart:
        raise ValueError(f"No profile/cart for customer: {customer_id}")

    decisions = artifact.get("decisions", [])
    tone_decision = next((d for d in decisions if d.get("type") == "tone"), None)
    tone = tone_decision.get("value", "warm + personal") if tone_decision else "warm + personal"

    agent = create_copywriter_agent(model)
    deps = CopywriterDeps(
        customer_id=customer_id,
        profile=profile,
        cart=cart,
        tone=tone,
        hero_image_url="/whiskers.jpg",
    )
    prompt = build_email_prompt(artifact, profile, cart)
    result = await agent.run(prompt, deps=deps)
    return result.output
The flow:
  • Extract customer ID and tone from Agent 1's decision artifact.
  • Load profile and cart from synthetic data (in production, this would be Lakebase).
  • Build CopywriterDeps with all context.
  • Call await agent.run(prompt, deps=deps) — Claude writes the email.
  • Return the parsed EmailContent (subject, body_html, body_text, CTA, etc.).

4. The Output Models

Pydantic models define what each agent outputs. These are the contracts between agents and the workflow.

4a. DecisionArtifact — What Agent 1 Produces

class Decision(BaseModel):
    type: str  # suppress_from, prioritize_in, tone, send_time, channel
    target: str | None = None
    value: str | None = None
    weight: float | None = None
    reason: str | None = None


class ContributingSignal(BaseModel):
    signal: str
    value: Any
    weight: float


class DecisionArtifact(BaseModel):
    decision_id: str
    customer_id: str
    journey_id: str
    trigger_event_id: str
    verdict: str
    decisions: list[Decision]
    contributing_signals: list[ContributingSignal]
    rationale: str
    trace_id: str | None = None
    created_at: datetime
DecisionArtifact is the structured output of Agent 1. It includes:
  • verdict — the recommendation (e.g., "re-prioritize", "proceed", "defer").
  • decisions — a list of specific actions: suppress campaigns, prioritize campaigns, set tone, pick send time, pick channel.
  • contributing_signals — which inputs influenced the decision and how much (weight 0-1).
  • rationale — one paragraph explaining the reasoning chain (human-readable).
  • IDs for tracing: decision_id, journey_id, trace_id (MLflow).

4b. EmailContent — What Agent 2 Produces

class EmailContent(BaseModel):
    subject: str
    body_html: str
    body_text: str
    hero_image_url: str
    cta_text: str
    cta_url: str
Agent 2 outputs the complete email payload:
  • subject — subject line (max 50 chars per the prompt).
  • body_html and body_text — the email content in both formats.
  • hero_image_url, cta_text, cta_url — the image and call-to-action.

5. How the Workflow Orchestrates Both Agents

The workflow lives in src/maestro/workflow.py. It's a single async DBOS function decorated with @DBOS.workflow(). It owns the lifecycle: create the event, run Agent 1, persist the decision, sleep, re-evaluate, run Agent 2, simulate send. If the server crashes during the sleep, DBOS picks up exactly where it left off.

5a. Helper Functions

            sslmode="require",

    from maestro.app import app
    if hasattr(app.state, "db_params") and app.state.db_params:
        return app.state.db_params
    from maestro.bootstrap import get_lakebase_conn_params as _cli_params
    return _cli_params()


def _get_model():
    """Get the module-level MODEL from app.py (already initialized)."""
    from maestro.app import MODEL
    return MODEL
These helpers solve two problems:
  • _get_db_params() — handles both local (SDK-based auth) and cloud (env-var-based) deployments.
  • _get_model() — retrieves the pre-initialized LLM model. We don't create a new one per workflow step; we reuse the module-level instance.

5b. Async DBOS Steps (for LLM Calls)

@DBOS.step()
async def run_agent_step(event_json: str) -> str:
    """Run the Maestro agent. Returns DecisionArtifact as JSON."""
    from maestro.agent import run_maestro

    event = CartAbandonedEvent.model_validate_json(event_json)
    model = _get_model()
    result = await run_maestro(event, model, None)
    return result.model_dump_json()


@DBOS.step()
async def compose_email_step(artifact_json: str) -> str:
    """Compose personalized email via copywriter agent. Returns EmailContent JSON."""
    from maestro.email_agent import compose_email

    model = _get_model()
    email = await compose_email(artifact_json, model)
    return email.model_dump_json()
@DBOS.step() marks a function as a durable step. DBOS:
  • Checkpoints after each step completes (not before).
  • Persists the checkpoint to Lakebase (the system database).
  • If the process crashes, DBOS rehydrates from the last checkpoint and resumes from the next step.
These two steps are marked async because they call LLMs (Claude via Pydantic AI). The result is always JSON — that way, the next step can deserialize it.

5c. Sync DBOS Steps (for Database Operations)

@DBOS.step()
def persist_decision_step(decision_json: str) -> str:
    """Persist DecisionArtifact to Lakebase decisions table."""
    import psycopg2

    decision = json.loads(decision_json)
    decision_id = decision.get("decision_id", f"dec_{uuid.uuid4().hex[:8]}")
    params = _get_db_params()

    conn = psycopg2.connect(**params)
    conn.autocommit = True
    cur = conn.cursor()
    cur.execute(
        """INSERT INTO decisions (decision_id, customer_id, journey_id,
            trigger_event_id, verdict, decisions, contributing_signals,
            rationale, trace_id, created_at)
        VALUES (%s, %s, %s, %s, %s, %s, %s, %s, %s, %s)
        ON CONFLICT (decision_id) DO UPDATE SET
            verdict = EXCLUDED.verdict,
            decisions = EXCLUDED.decisions,
            contributing_signals = EXCLUDED.contributing_signals,
            rationale = EXCLUDED.rationale""",
        (
            decision_id,
            decision["customer_id"],
            decision.get("journey_id", ""),
            decision.get("trigger_event_id", ""),
            decision["verdict"],
            json.dumps(decision.get("decisions", [])),
            json.dumps(decision.get("contributing_signals", [])),
            decision.get("rationale", ""),
            decision.get("trace_id"),
            decision.get("created_at", datetime.now(CT).isoformat()),
        ),
    )
    cur.close()
    conn.close()
    return decision_id
Even sync database operations get the @DBOS.step() decorator. This step:
  • Parses the decision JSON (from Agent 1).
  • Inserts (or updates) the decision in the decisions table.
  • Returns the decision_id for the next steps to reference.
If the workflow crashes after this step, DBOS knows it's already persisted and won't repeat it.

5d. The Unified Workflow — Everything Together

@DBOS.workflow()
async def unified_journey_workflow(event_json: str, delay_seconds: int = 15) -> str:
    """Complete Beat 2 + 2.5 + 3 workflow (async).

    Steps:
      1. Agent reasoning -> DecisionArtifact
      2. Persist decision to Lakebase
      3. Save journey state
      4. Durable sleep (simulates optimal send window)
      5. Re-evaluate context (check for changes)
      6. Compose email (copywriter agent)
      7. Simulate send (write to sent_emails)

    Returns: JSON dict with journey_id, decision_id, email_id, evaluation
    """
    # ── Step 1: Agent reasoning ────────────────────────────────────────
    decision_json = await run_agent_step(event_json)
    decision = json.loads(decision_json)
    customer_id = decision.get("customer_id", "unknown")

    # ── Step 2: Save journey state (before decision — FK constraint) ──
    journey_id = decision.get(
        "journey_id",
        f"jrn_{customer_id}_{uuid.uuid4().hex[:6]}",
    )
    send_time_decisions = [
        d for d in decision.get("decisions", []) if d.get("type") == "send_time"
    ]
    due_ts = (
        send_time_decisions[0].get("value", datetime.now(CT).isoformat())
        if send_time_decisions
        else datetime.now(CT).isoformat()
    )
    save_journey_step(
        journey_id=journey_id,
        customer_id=customer_id,
        step="awaiting_send",
        due_ts=due_ts,
        state_blob=decision_json,
    )

    # ── Step 3: Persist decision ───────────────────────────────────────
    decision_id = persist_decision_step(decision_json)

    # ── Step 4: Durable sleep ──────────────────────────────────────────
    await DBOS.sleep_async(delay_seconds)

    # ── Step 5: Re-evaluate context ────────────────────────────────────
    eval_json = re_evaluate_step(journey_id, decision_json)
    evaluation = json.loads(eval_json)

    # ── Step 6: Compose email (if proceed/adjust) ──────────────────────
    email_json = None
    email_id = None
    if evaluation["action"] in ("proceed", "adjust"):
        context = (
            decision_json
            if evaluation["action"] == "proceed"
            else evaluation.get("updated_artifact", decision_json)
        )
        email_json = await compose_email_step(context)

        # ── Step 7: Simulate send ──────────────────────────────────────
        email_id = simulate_send_step(journey_id, customer_id, email_json)
        update_journey_status_step(journey_id, "completed")
    else:
        update_journey_status_step(journey_id, "failed")

    return json.dumps({
        "journey_id": journey_id,
        "decision_id": decision_id,
        "email_id": email_id,
        "evaluation": evaluation["action"],
    })
This is the heart of the system. Let's trace it: Step 1: Agent Reasoning await run_agent_step(event_json) calls Agent 1. It's async, so the event loop doesn't block. Claude reasons, makes tool calls, and returns a DecisionArtifact as JSON. Step 2: Save Journey State Before we persist the decision, we extract the send_time from the decision (which Agent 1 produced). This becomes the due_ts for when we should wake up. We save it to the journey_state table, which tracks the lifecycle. Step 3: Persist Decision The decision artifact is saved to the decisions table. Step 4: Durable Sleep await DBOS.sleep_async(delay_seconds) is the magic. DBOS persists the sleep deadline to Lakebase. If the server crashes, DBOS knows to wake up at that time. This is how the workflow survives restarts. Step 5: Re-evaluate Context After waking up, we check if anything changed (customer moved to a different segment, another campaign triggered, etc.). If changes are detected, the decision might be adjusted. Steps 6-7: Compose Email & Send If the re-evaluation says "proceed" or "adjust", we run Agent 2 (the copywriter). It produces an EmailContent. Then we simulate sending (write to the sent_emails table). Return Value The workflow returns a JSON object with all the IDs for downstream tracking.

6. How It All Gets Triggered

The API lives in src/maestro/app.py. Two endpoints kick off the agent loop — one for testing, one for durable execution.

6a. The /api/run Endpoint — Instant Agent Execution

@app.post("/api/run")
async def run_agent():
    """Run Beat 2 agent — async endpoint, works with DBOS event loop."""
    import traceback
    try:
        from maestro.agent import run_maestro
        from maestro.synthetic import CINDY_CAMPAIGNS, CINDY_EVENT

        start = time.perf_counter()
        result = await run_maestro(CINDY_EVENT, MODEL)
        elapsed = time.perf_counter() - start
    except Exception as e:
        raise HTTPException(status_code=500, detail=f"Failed: {e}\n{traceback.format_exc()[-500:]}")

    suppress_targets = {d.target for d in result.decisions if d.type == "suppress_from"}
    prioritize_targets = {d.target for d in result.decisions if d.type == "prioritize_in"}
    campaigns = []
    for c in CINDY_CAMPAIGNS:
        st = c.status
        if c.campaign_id in suppress_targets or c.name in suppress_targets:
            st = "suppressed"
        elif c.campaign_id in prioritize_targets or c.name in prioritize_targets:
            st = "prioritized"
        campaigns.append({"name": c.name, "campaign_id": c.campaign_id, "status": st})

    return {"artifact": result.model_dump(), "campaigns": campaigns, "latency_s": elapsed}
/api/run is for testing and UI panels. It:
  • Calls Agent 1 directly (not through the workflow).
  • Returns the DecisionArtifact immediately.
  • Also enriches the response with campaign statuses (suppressed/prioritized) for the UI.
  • Measures latency so we know if Agent 1 is staying under the 2-second ceiling.
This is not durable — if the server crashes during the LLM call, the request is lost.

6b. The /api/workflow Endpoint — Durable Workflow Execution

@app.post("/api/workflow")
async def start_unified_workflow(body: dict):
    """Start unified Beat 2+2.5+3 workflow: agent -> persist -> sleep -> re-eval -> email -> send."""
    import random

    from maestro.synthetic import CINDY_EVENT
    from maestro.workflow import unified_journey_workflow

    event_json = CINDY_EVENT.model_dump_json()
    delay = body.get("delay", random.randint(15, 20))

    handle = await DBOS.start_workflow_async(unified_journey_workflow, event_json, delay)
    return {
        "workflow_id": handle.get_workflow_id(),
        "delay": delay,
        "status": "started",
    }
/api/workflow is for production durable execution. It:
  • Calls DBOS.start_workflow_async(unified_journey_workflow, event_json, delay).
  • DBOS handles all the checkpointing, recovery, and retries.
  • Returns immediately with a workflow_id.
  • The frontend can poll /api/workflow/{workflow_id}/phases to track progress.
This is durable — the entire workflow survives server crashes.

7. The Key Insight

The two agents never talk to each other directly.

Agent 1 produces a DecisionArtifact — a structured record of what to do, why, and when.

That artifact gets persisted to Lakebase. After the durable sleep, the artifact is loaded and passed to Agent 2 as context.

The DBOS workflow is the glue — it owns the lifecycle, the timing, and the recovery. If the server restarts during step 4 (the sleep), DBOS wakes up at the right time and resumes at step 5, with the artifact already persisted and available.

This decoupling is intentional. Agent 1 is a reasoner (high latency, high stakes). Agent 2 is a writer (low latency, deterministic). Splitting them lets us:

  • Test each agent independently.
  • Swap Agent 2's implementation without touching Agent 1.
  • Inject delays between reasoning and writing (simulating the optimal send window).
  • Re-evaluate context between the two agents (checking for last-minute changes).

src/maestro/agent.py Agent 1: reasoning agent, 8 tools, system prompt, runner
src/maestro/email_agent.py Agent 2: copywriter agent, no tools, prompt templates
src/maestro/workflow.py DBOS workflow orchestrating both agents, persistence, sleep, re-evaluation
src/maestro/models.py Pydantic data models: DecisionArtifact, EmailContent, all tool return types
src/maestro/app.py FastAPI app, model init, /api/run (instant), /api/workflow (durable)

Next steps: Read the files in the order above. Start with models.py to understand the data shapes, then agent.py to see how Agent 1 reasons, then email_agent.py, then workflow.py to see how DBOS orchestrates, then app.py to see how the API triggers it all.
